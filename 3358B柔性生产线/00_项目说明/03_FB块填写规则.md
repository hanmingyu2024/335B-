# FB块填写规则

## 1. FB 的原则

`FB` 有实例记忆，配实例 `DB`。  
所以 FB 的接口设计最关键。

你这个项目里，下面这些都建议按 `FB + 实例DB` 做：

- `FB_StationCtrl`
- `FB_Comm`
- `FB_Alarm`
- `FB_Cylinder`
- `FB_Motor`
- `FB_Conveyor`
- `FB_Sensor`

## 2. FB 与实例 DB 的关系

在博途中：

- `Input / Output / InOut / Static`
  都属于 `FB` 的定义
- 实例 `DB` 会根据这些内容自动展开

所以：

**实例 DB 的变量格式不是先填 DB，而是先填 FB。**

## 3. Input / Output / InOut / Static 怎么分

### Input

只读输入参数，比如：

- 命令
- 反馈
- 时间参数
- 外部模式信号

### Output

块对外给出的结果，比如：

- 到位正常
- 故障
- 准备好
- 运行中

### InOut

双向结构体，适合：

- `stStation`
- `stComm`
- `stAlarm`

### Static

块内部长期保存的状态，比如：

- 步号
- 上一扫描状态
- 计数器
- 定时器实例
- 记忆标志

## 4. 当前项目的推荐思路

### 站控类 FB

优先推荐：

- `Input`：参数结构、外部命令
- `InOut`：站状态结构
- `Static`：步序状态、TON 定时器、上次状态

### 设备类 FB

优先推荐：

- `Input`：命令、反馈、超时参数
- `Output`：动作输出、完成状态、故障状态
- `Static`：定时器、锁存位

## 5. 不建议的做法

- 所有变量都塞进 `Input`
- 把需要长期记忆的状态写成 `Temp`
- 先手填实例 DB，再回头补 FB 接口

## 6. 结论

对于 FB：

- 必须先设计接口
- `Static` 非常关键
- 实例 DB 自动跟随 FB 生成
